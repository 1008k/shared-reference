# エージェント拡張の導入判断

この文書は、常時ルール、パス限定ルール、skill、subagent、hook、automation、MCPの置き場所を判断する補助文書です。
外部接続や決定的な強制が必要かを分けつつ、同じ粒度で比較できるようにします。

## いつ検討するか
- 同じ依頼手順を複数回説明しているとき
- 文書更新やレビュー観点が繰り返し登場するとき
- 単なる短いルール列では足りず、手順、例、補助スクリプト参照が必要なとき
- そのリポジトリ固有の進め方を、以後のセッションでも再利用したいとき

## skillに向くもの
- 複数手順からなる定型ワークフロー
- リポジトリ固有の文書読み順や判断基準
- 例、テンプレート、補助スクリプトを伴う作業
- 明示呼び出しでも暗黙選択でも価値がある作業

## skillに向かないもの
- 1行で済む恒久ルール
- `AGENTS.md` に短く書けば足りる指示
- 実際には1回しか使わない手順
- 外部接続や認証が本体で、ワークフロー自体は薄いもの

## `AGENTS.md` とskillの切り分け
- `AGENTS.md`: 常に効かせたい短い恒久ルール
- skill: 条件に応じて使う再利用ワークフロー

## パス限定ルールとの切り分け
- ルートのルール: リポジトリ全体で常に必要な前提や規約
- パス限定ルール: 特定のディレクトリやファイル種別だけに適用する制約
- モノレポや責務の異なる領域ができるまでは、専用ルールファイルを先回りして増やさない

## skillとsubagentの切り分け
- skill: 親セッション内で手順を確認しながら進めたい再利用ワークフロー
- subagent: 深い調査、大量ログ解析、依存監査など、途中結果を親セッションから隔離したい副作業
- 最終結果だけで判断できる独立作業はsubagent、途中で利用者が調整したい作業はskillを優先する

## skillとautomationの切り分け
- skill: 手順や判断基準そのものを再利用する
- automation: 安定した手順を定期実行する
- まず手動で安定させ、繰り返し価値が見えたら skill にし、その後で automation を検討する

## hookやpermissionとの切り分け
- 文章のルール: 状況に応じた判断や方針をエージェントへ伝える
- hook、permission、CI: 禁止操作、編集後の検査、必須チェックなどを決定的に強制する
- 「必ず実行する」「絶対に禁止する」が重要な境界では、指示文だけに依存しない
- 導入時は、対象イベント、失敗時の扱い、必要権限、外部通信、ローカル設定と共有設定の境界を明確にする

## MCPとskillの切り分け
- MCP: 外部システムへの接続やツール提供
- skill: そのツールをどういう順で使うかの手順
- 外部システム利用が必要でも、手順の再利用価値が高ければ `skill + MCP` を検討する

## 判断観点
- 再利用性: 別の依頼でも同じ流れで使えるか
- リポジトリ依存度: このリポジトリの文書や構造に強く依存するか
- 説明量: `AGENTS.md` だけでは収まらないか
- コンテキスト: 常時読み込む必要があるか、条件付きまたは隔離実行で足りるか
- 強制力: 推奨や判断基準か、機械的に保証すべき制約か
- 保守コスト: 更新箇所が増えすぎないか
- 誤誘導リスク: 曖昧なdescriptionで意図しない場面でも呼ばれないか

## 導入の目安
- 同種の作業で2回以上ほぼ同じ指示を書いている
- 手順を固定した方がミスや読み過ぎを減らせる
- リポジトリ内に置いた方が、外部参照より判断が速い

## 記録先
- リポジトリ固有skillは `.agents/skills/<skill-name>/SKILL.md`
- skill追加の背景や補足判断は、必要ならこのディレクトリや `docs/tracks/` に残す
- 短い恒久ルールへ落とし込める部分は、必要に応じて `AGENTS.md` へ戻す
