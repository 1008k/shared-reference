# UI/UXルール

この文書は、プロジェクト横断で使うユーザー向けUIの設計、実装、レビュー時の基本ルールです。各プロジェクトでは、ローカルの `docs/rules-ux.md` を入口にし、この共有ルールを基本としてプロジェクト固有のデザイン方向、画面パターン、例外を追加します。

## 基本方針
- 単に機能を並べるのでなく、ユーザーが迷わず目的を達成できる状態を優先する。
- UIを持つ作業では、実装前にユーザーの目的、最短の操作手順、主要な状態を短く確認する。
- 小さな文言修正、余白調整、既存UIの局所修正では、この文書の観点を必要な分だけ使う。
- UIは足すより削ることを優先し、主要目的に直接関係しない要素は後回しにする。
- 見た目の印象より、操作の分かりやすさ、予測可能性、回復しやすさを優先する。
- プロジェクト固有のデザイン文書は、この文書のUX原則を置き換えず、見た目と画面パターンを揃える補助として扱う。

## 体験品質
- 成果物の品質は、機能が動くことだけでなく、気持ちよく使える見た目、操作感、情報設計、言葉の自然さを含めて判断する。
- UIを持つ成果物では、主要導線を実際に触ったときの迷い、待ち時間、不安、視線移動、誤操作のしやすさを確認する。
- デザインは装飾ではなく、ユーザーが状況を理解し、次の行動を自然に選べるようにするための設計として扱う。
- 既存デザインやブランドがある場合はそれに合わせる。ない場合も、余白、階層、コントラスト、配置、文言、動きの一貫性を整える。

## UX原則
- ユーザーが考え込まずに操作できることを最優先にする。
- 1画面には原則として1つの主目的を持たせる。
- 操作結果は予測可能にし、意外な挙動を避ける。
- ユーザーの記憶に頼らず、ラベル、配置、状態表示によってUI側で理解を助ける。
- 入力や操作には、処理中、成功、失敗などのフィードバックをできるだけ早く返す。
- ブラウザ向けUIでは、意味に合ったHTML、標準の対話要素、既定のフォーカス移動を活かし、見た目のために基本的な操作性を壊さない。
- Web UIでは、セマンティックHTML、キーボード操作、見えるフォーカス、適切な代替テキスト、フォームラベル、色だけに依存しない状態表現を基本確認に含める。
- 新しいWeb UI機能を使う場合も、非対応環境で主要導線が破綻しないことを優先し、段階的改善として扱う。

## 実装前の確認
UIを新規作成または大きく変更する前に、次の観点を短く確認する。

1. User Goal: ユーザーがその画面で達成したいこと。
2. User Flow: 最短で目的に到達する操作ステップ。
3. UI Structure: 画面のどこに何を配置するか。
4. Edge Cases: ローディング、空状態、エラー時の扱い。
5. Design Direction: プロジェクト固有のデザイン方向から外れていないか。
6. Modern Web Fit: ブラウザ標準の機能でより単純に実現できないか。

この確認は、長い出力フォーマットではなく設計漏れを防ぐチェックとして扱う。画面追加、導線変更、フォーム追加など影響が大きい場合だけ、判断理由や画面構成を記録する。

## アクション
- 主要アクションは、画面または明確な操作領域ごとに1つまでにする。
- 主要アクション以外は、セカンダリ、リンク、メニュー、補助操作として視覚的な優先度を下げる。
- ボタン文言は、できるだけ動詞ベースで具体的にする。
- 汎用的な「保存」より、ユーザーの結果が分かる「記事を公開」「招待を送信」のような文言を優先する。

## フォーム
- 入力項目は、目的達成に必要な最小限にする。
- バリデーションは、入力中に邪魔にならないタイミングで行う。
- エラーは、原因と解決方法を近い場所に表示する。
- 必須項目、入力形式、制約は、ユーザーが送信前に推測しなくてよいように示す。
- 入力フォームは、意味のあるまとまりで整理し、過剰に分割しない。

## レイアウト
- 視線の流れを意識し、主要情報、主要アクション、補助情報の順序を明確にする。
- 情報を詰め込みすぎず、余白、見出し、グルーピングで読み取りやすくする。
- 同じ意味の操作や情報は、一貫した位置、見た目、文言で扱う。
- 初見ユーザーが最初に見るべき情報と、慣れたユーザーが素早く使う情報を混同しない。

## 状態設計
ユーザー向けUIでは、次の状態を設計に含める。

- ローディング状態: 何が処理中か、待てばよいのか、操作できるのかを示す。
- 空状態: データがない理由と、次に取れる自然な行動を示す。
- エラー状態: 何が起きたか、ユーザーが直せるか、再試行できるかを示す。

状態表示は、単にメッセージを出すだけでなく、主要アクションの無効化、再試行導線、入力値の保持など、操作継続のしやすさも含めて設計する。

## ブラウザ機能の扱い
- モーダル、ツールチップ、開閉UI、フォーム補助などは、独自の複雑な再実装より、`dialog`、Popover API、適切なフォーム要素など標準機能で十分かを先に確認する。
- レイアウトや部品の適応は、必要に応じてcontainer queriesなど現代的なCSSを検討してよいが、可読性や保守性が下がる場合は単純な構成を優先する。
- アニメーションや視覚効果は、主目的の理解を助ける範囲に留め、応答性や読みやすさを損なわないようにする。

## 避けること
- モーダルを多用しない。
- 不要な確認ダイアログを出さない。
- 主目的が曖昧な画面を作らない。
- 入力フォームを過剰に分割しない。
- 説明文だけで使い方を補い、UI自体の分かりにくさを放置しない。

## 例外と調整
- ダッシュボード、管理画面、一覧画面では、1画面に複数の補助目的が並ぶことがある。その場合も、画面全体の主目的と、領域ごとの主要アクションを明確にする。
- テーブルやカード一覧では、行やカードごとに同じ操作が繰り返されることがある。この場合は、画面全体で1つというより、操作領域ごとの優先度で判断する。
- 業務アプリでは余白を広げすぎると一覧性が落ちることがある。情報密度と操作速度を優先すべき場面では、整理された密度を許容する。
- 破壊的操作、支払い、外部送信など取り消しにくい操作では、確認を省きすぎない。確認が必要な場合も、内容と影響範囲を具体的に示す。

## UXセルフレビュー
実装完了前に、変更の大きさに応じて次を確認する。

- 初見ユーザーでも迷わないか。
- 操作ステップは最小か。
- 入力項目は多すぎないか。
- 主要アクションは明確か。
- ローディング、空状態、エラー状態が考慮されているか。
- エラー時に、原因と次の行動が分かるか。
- 見た目、文言、操作感、状態変化が気持ちよく一貫しているか。
