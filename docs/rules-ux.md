# UI/UXルール

プロジェクト横断で使うユーザー向けUIの設計、実装、レビュー時の基本ルールです。各プロジェクトではローカルの `docs/rules-ux.md` を入口にし、固有のデザイン方向、画面パターン、例外だけを追加します。

## 基本方針
- ユーザーが迷わず目的を達成できることを優先し、主要目的に直接関係しない要素は後回しにする。
- 見た目は体験品質の一部として扱い、情報設計、操作感、文言、状態変化の一貫性を整える。
- 既存デザインやブランドがある場合は合わせ、ない場合も余白、階層、コントラスト、配置、文言の整合を保つ。
- 小さな文言修正や局所調整では、必要な観点だけを使う。

## 設計前に見ること
- User Goal: その画面でユーザーが達成したいこと。
- User Flow: 最短で目的に到達する操作ステップ。
- UI Structure: 主要情報、主要アクション、補助情報の配置。
- States: ローディング、空状態、エラー、成功、無効化の扱い。
- Design Fit: プロジェクト固有の見た目や画面パターンから外れていないか。
- Web Fit: 標準HTMLやブラウザ機能でより単純に実現できないか。

## 操作とフォーム
- 主要アクションは、画面または明確な操作領域ごとに絞り、補助操作は視覚的な優先度を下げる。
- ボタンやリンクの文言は、ユーザーに起きる結果が分かる具体的な動詞にする。
- 入力項目は目的達成に必要な最小限にし、意味のあるまとまりで整理する。
- エラー、必須項目、入力形式、制約は、ユーザーが原因と次の行動を近い場所で理解できるように示す。

## レイアウトと状態
- 初見ユーザーが最初に見る情報と、慣れたユーザーが素早く使う情報を混同しない。
- 同じ意味の操作や情報は、一貫した位置、見た目、文言で扱う。
- ローディング、空状態、エラー状態では、何が起きているか、待てばよいか、再試行できるか、次に何をすればよいかを示し、入力値の保持、主要アクションの無効化、再試行導線まで含めて考える。

## アクセシビリティとWeb機能
- 意味に合ったHTML、標準の対話要素、キーボード操作、見えるフォーカス、フォームラベル、代替テキスト、色だけに依存しない状態表現を基本確認に含める。
- Web UIは、拡大表示、文字サイズ、動きを減らす設定でも、情報や主要操作が隠れず利用できることを確認する。
- モーダル、ツールチップ、開閉UI、フォーム補助は、独自実装より `dialog`、Popover API、適切なフォーム要素などで足りるかを先に確認する。
- アニメーションや視覚効果は、主目的の理解を助ける範囲に留め、応答性や読みやすさを損なわない。

## 例外とレビュー
- ダッシュボード、管理画面、一覧画面では情報密度を許容してよいが、画面全体の主目的と領域ごとの主要アクションは明確にする。
- 破壊的操作、支払い、外部送信など取り消しにくい操作では、確認を省きすぎず、内容と影響範囲を具体的に示す。
- 完了前に、初見で迷わず最小の操作で目的を達成でき、入力が過剰でなく、通常・空・失敗の各状態で状態表示、主要アクション、次の行動が明確かを確認する。
