# 文書表記ルール

この文書は、プロジェクト横断で使う文書表記ルールです。各プロジェクトでは、ローカルの `docs/rules-writing.md` を入口にし、この共有ルールを基本として固有の用語、文体、公開物の制約を追加します。

## 基本方針
- 表記はプロジェクトの既存文書と対象読者に合わせ、同じ種類の文書内で一貫させる。
- 日本語文書で英単語、英数字、記号を扱う場合は、プロジェクト既存の方針を優先する。明示方針がない場合は、機械的な空白追加を避け、読みやすさが落ちる箇所だけ語の選び方やバッククォートで補う。
- 例: `AIエージェント`, `Webアプリ`, `README.mdを参照する`
- コマンド、パス、設定キー、識別子、コード断片、複合英語の原表記は、必要に応じてバッククォートで囲んで保持する。
- 例: `npm test`, `config.json`, `feature flag`

## 用語統一
- 同じ対象を指す語は、同一文書内と関連文書間で表記をそろえる。
- 既存文書に正本となる表記がある場合は、それに合わせる。

## 文体
- 文書単位で文体をそろえ、`です・ます` と `である` を混在させない。
- 箇条書きの語尾は、同じ節の中でできるだけそろえる。

## 読者と文書種別
- READMEや導入手順は、初めて読む人が迷わず試せる順序で書く。
- 仕様書や設計メモは、決定事項、前提、制約、未決事項を分け、後から判断理由を追えるようにする。
- 運用メモやトラブルシュートは、症状、原因、確認方法、対応手順を近い場所にまとめる。
- エージェント向けプロンプトやルールは、命令、判断基準、完了条件を分け、長い背景説明を恒久ルールに混ぜない。

## 曖昧さを避ける
- `適宜`, `必要に応じて`, `適切に` などの曖昧な表現は、多用しない。
- 判断を委ねる場合は、判断基準をできるだけ併記する。

## Markdown文書の読みやすさ
- リンクテキストは、リンク先の内容が分かる表現にする。
- 例: `[公式ドキュメント](url)`, `[設定例](url)`
- `[ここをクリック](url)` やURLをそのままラベルにしたリンクは、文脈上必要な場合を除き避ける。
- 太字は重要語や判断軸に絞り、箇条書きの全項目を機械的に太字にしない。
- 箇条書きで `- **ラベル**: 本文` の形を多用しない。見出し化が自然な粒度なら、短い見出しと本文に分ける。
- コードブロックの前後には空行を入れ、本文、リスト、コードの境界を読み取りやすくする。

## 時間表現
- `最近`, `現在`, `今日では`, `最新` などの相対的な時間表現は、読む時点で意味が変わるため多用しない。
- 時間が判断に影響する場合は、`2026年6月時点` や `2026年5月発表` のように基準時期を併記する。
- 作業ログや近況メモなど、リアルタイム性そのものが意味を持つ文書では相対表現を使ってよい。

## AI生成っぽさを避ける
- AIっぽい定型表現を削るだけで終わらせず、具体的な事実、判断、制約、次の行動に置き換える。
- `いかがでしたか`, `ぜひ試してみてください`, `今後の展開が注目されます` のような定型的な締めは避ける。
- `重要`, `便利`, `高機能`, `シームレス` などの評価語だけで止めず、そう判断できる理由や場面を短く添える。
- `専門家によると`, `多くのユーザーが指摘している` などの曖昧な出典を残さない。
- 根拠を書く場合は、公式情報、仕様、Issue、検証結果などを具体化する。
- 測定していない数値、実在しない体験、未確認の効果は書かない。

## Changelog / release note
- `CHANGELOG.md` やrelease noteは、配布物の動作に関わる利用者向け変更だけを記録する。
- 各項目は1行ずつ簡潔に書き、何が変わったかがすぐ分かる表現にする。
- 内部実装詳細、文書調整、理由説明、実装方針、内部事情は、利用者への影響を理解するために必要な場合を除き含めない。
- 背景や判断理由を補足したい場合は、本文を長くせず、作業ログや設計メモの該当項目を参照する。
