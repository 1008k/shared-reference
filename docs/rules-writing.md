# 文書表記ルール

プロジェクト横断で使う文書表記ルールです。各プロジェクトではローカルの `docs/rules-writing.md` を入口にし、固有の用語、文体、公開物の制約だけを追加します。

## 基本方針
- 表記は既存文書と対象読者に合わせ、同じ種類の文書内で一貫させる。
- 同じ対象を指す語は、関連文書間で表記をそろえる。正本がある場合はそれに合わせる。
- 文書単位で文体をそろえ、`です・ます` と `である` を混在させない。
- 特別な理由がない限り、基本的に自然でやわらかいトーンにする。
- コマンド、パス、設定キー、識別子、コード断片、原表記を保ちたい語は、必要に応じてバッククォートで囲む。

## 読者と文書種別
- READMEや導入手順は、初めて読む人が迷わず試せる順序で書く。
- 仕様書や設計メモは、決定事項、前提、制約、未決事項を分け、後から判断理由を追えるようにする。
- 運用メモやトラブルシュートは、症状、原因、確認方法、対応手順を近い場所にまとめる。
- エージェント向けプロンプトやルールは、命令、判断基準、完了条件を分け、長い背景説明を恒久ルールに混ぜない。

## Markdown
- リンクテキストはリンク先の内容が分かる表現にする。
- 太字は重要語や判断軸に絞り、箇条書きの全項目を機械的に太字にしない。
- `- **ラベル**: 本文` を多用せず、見出し化が自然な粒度なら短い見出しと本文に分ける。
- コードブロックの前後には空行を入れ、本文、リスト、コードの境界を読み取りやすくする。

## 曖昧さと時間表現
- `適宜`, `必要に応じて`, `適切に` などを多用せず、判断を委ねる場合は基準を併記する。
- `最近`, `現在`, `今日では`, `最新` などは読む時点で意味が変わるため多用しない。
- 時間が判断に影響する場合は、基準時期を併記する。作業ログや近況メモでは相対表現を使ってよい。

## AI生成っぽさを避ける
- 定型表現を削るだけで終わらせず、具体的な事実、判断、制約、次の行動に置き換える。
- 仰々しい表現、大げさな比喩、過度に劇的な言い回しは避け、落ち着いた自然な表現にする。
- 評価語だけで止めず、そう判断できる理由や場面を短く添える。
- 曖昧な出典、測定していない数値、実在しない体験、未確認の効果を残さない。
- 根拠を書く場合は、公式情報、仕様、Issue、検証結果などを具体化する。

## Changelog / release note
- `CHANGELOG.md` やrelease noteは、配布物の動作に関わる利用者向け変更だけを記録する。
- 各項目は1行ずつ簡潔に書き、何が変わったかがすぐ分かる表現にする。
- 内部事情や理由説明は、利用者への影響を理解するために必要な場合を除き含めない。
- 背景や判断理由を補足したい場合は、本文を長くせず、作業ログや設計メモの該当項目を参照する。
