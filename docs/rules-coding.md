# コーディングルール

この文書は、プロジェクト横断で使う実装判断の基本ルールです。各プロジェクトでは、ローカルの `docs/rules-coding.md` を入口にし、この共有ルールを基本としてプロジェクト固有の上書きや例外を追加します。

## 基本方針
- 変更は最小差分で行い、影響範囲を明確にする。
- 可読性、保守性、セキュリティを優先する。
- 仕様判断は各プロジェクトの仕様正本を優先し、日常運用は各プロジェクトのREADMEに合わせる。
- 実装と仕様が衝突した場合や、不具合の原因や期待挙動が曖昧な場合は、実装、仕様、運用のどこを更新すべきか整理してから直す。
- 技術固有のルールは、採用スタックが決まってから必要最小限で追加する。
- 採用した製品、プラットフォーム、フレームワークに公式または広く受け入れられたベストプラクティスがある場合は、正当な理由がない限りそれに沿って設計・実装する。
- UI、データ処理、認証、決済、パーサーなど実装リスクが高い領域は、実績のあるフレームワーク、ライブラリ、標準機能の活用を先に検討する。
- 依存追加による複雑化や制約が要件に見合わない場合は、保守性、拡張性、学習コスト、置き換えやすさを踏まえて手組み実装を選んでよい。
- 外部ライブラリ、フレームワーク、外部APIの仕様確認が必要な場合は、一次情報または現在の公式ドキュメントを優先する。

## 共有ルールの扱い
- 共有ルールには、複数プロジェクトで繰り返し使い、特定スタックや個別仕様に依存しない判断だけを置く。
- プロジェクト固有の要件、互換性、ドメイン知識、公開仕様、運用制約は、共有ルールではなく各プロジェクトの文書に置く。
- 共有ルールを追加する前に、既存ルールやskillで代替できないか、命令が強すぎないか、削除や縮約の条件があるかを確認する。
- 例外や上書きが必要な場合は、共有側を弱めるより、各プロジェクトのローカルルールで理由を添えて補足する。

## スタック選定
- 実装スタックや開発基盤は、成熟度、保守性、互換性を満たす範囲で、速度、導入の軽さ、標準機能、置き換えやすさのバランスが良い選択肢を優先する。
- 既存の慣習だけで技術を固定せず、現在の要件に対してより単純で開発体験の良い候補があるなら先に比較する。
- 共有ルールでは、特定の実装スタックを標準として固定しない。
- 各プロジェクトでスタックが決まったら、必要最小限の技術固有ルールだけをローカルに追加する。

## Web UIの既定
- ブラウザ向けUIを持つ案件では、できるだけ標準機能を優先し、古い慣習的な独自実装より、現在の主要ブラウザで成熟したWeb Platform APIを先に検討する。
- ブラウザ実装の判断では、アクセシビリティ、パフォーマンス、セキュリティを機能追加と同列の要件として扱う。
- 対象ブラウザが固定されていない場合は、まず広く利用可能な機能を優先し、採用根拠の確認にはBaselineや各仕様の一次情報を使う。
- モダンなWeb機能を採用する場合も、非対応環境で致命的に壊さないことを優先し、段階的改善または安全なフォールバックを検討する。

## 設計と責務
- 複数箇所で共有される責務は、正本となる振る舞いを一つ定義して一貫して扱う。
- プラットフォーム差分や外部連携差分は適応層で吸収し、中心ロジックを分岐させない。
- 入出力、外部依存、ドメインロジックを分離し、中心ロジックを環境依存にしない。
- 同じ意味の値や判定、設定、既定値、feature flagは散在させず、定義箇所を集約する。
- 分離や共通化は、責務境界と読みやすさが良くなるときに行い、まだ一箇所でしか使わず見通しもよい処理は無理に別ファイル化しない。
- 例外ケースを足す前に、責務分割やデータ構造の見直しで吸収できないかを先に検討する。
- 一時対応や互換処理を入れる場合は、対症療法で固定化させず、削除条件や見直し時期を明示する。

## 実装ルール
- 処理の分割は、意図や境界が明確になる単位で行う。
- 一連の流れを追う方が読みやすい処理や、単発で文脈依存が強い短い処理は呼び出し元の近くに置く。
- エラーは握りつぶさず、ログまたは返却値で明示する。
- 外部入力は境界ごとにバリデーションを行う。
- 命名は一貫性を優先する。
- 生成物やビルド成果物は直接編集しない。

## テスト/品質
- 重要ロジックにはテストを追加する。
- 確認範囲は変更のリスクに合わせる。文言や局所的な見た目の変更は軽く、共有ロジック、保存形式、認証、決済、公開導線、外部連携に触れる変更は広めに確認する。
- ビルドやテストが導入済みなら、変更時は最低限それらを実行してから共有する。
- 文言修正、表現の明確化、コメント更新など、振る舞いに影響しない変更では、導入済みのビルドとテストを省略してよい。
- CIを導入したら、可能な範囲で `lint / test / build` を自動化する。
- テストランナーやcoverage設定は、採用スタックに合わせて定義する。
- Web UIを持つ案件では、必要に応じてLighthouse、実ブラウザ確認、ユーザーフローに沿った軽量な計測を組み合わせ、主要導線のアクセシビリティ、表示性能、操作応答を確認する。

## セキュリティ
- 機密情報をリポジトリに保存しない。
- 秘密情報をコードに埋め込まない。
- 機密情報は、採用した実行環境やホスティング基盤が提供する秘密情報管理機構で扱う。
- ログ、エラーメッセージ、デバッグ出力に機密情報や個人情報をそのまま出さない。
- 外部入力は、境界ごとに型、形式、長さ、許可値を意識して検証する。
- 依存更新時は脆弱性情報を確認する。
- 依存追加時は、必要性、保守状況、権限範囲を確認する。
- 外部通信先や権限追加はレビュー対象にする。
- 外部通信や外部サービス追加時は、必要性と影響範囲を明記する。

## 技術固有ルールの追加
- 言語、ランタイム、フレームワーク固有のルールは、採用後に必要なものだけ追加・更新する。
- 使わない技術の節は増やさず、不要なら書かない。

## 運用
- 公開用の成果物が必要な場合は、再現できる手順で作成し、可能ならCIで作成・配布する。
- 繰り返し発生する検証、変換、デプロイ、文書チェックは、保守負荷が見合う範囲でCIサービスやスクリプトなどによる自動化を優先して検討する。
- 軽微な修正では作業ログを必須にしないが、仕様、設計、運用、公開物に影響する大きめの変更では作業文脈を残してから進めてよい。
- ローカル専用の生成物、一時ファイル、検証用ファイル、個人用補助スクリプトは、明示的な理由がない限り含めない。
